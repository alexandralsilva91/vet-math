# Vet Math – Project Context

## Project Overview

**Vet Math** is a professional veterinary medical calculator designed primarily for small animal practitioners. Although the initial version is intended for personal use, the project must be built from the beginning as a scalable, production-quality application capable of becoming a commercial product.

The application should prioritize:

* Accuracy
* Simplicity
* Speed
* Excellent user experience
* Maintainable architecture
* Scalability
* Professional software engineering practices

Whenever making architectural decisions, prioritize solutions that are easy to understand and maintain, since the project owner is still learning software development.

# My Protocols

Users should be able to:

* Create protocols
* Edit protocols
* Delete protocols
* Duplicate protocols
* Favorite protocols
* Organize protocols
* Add multiple drugs
* Save notes

Each protocol contains multiple drugs.

Each drug automatically calculates according to patient body weight.

Protocols are private to each user.

Future versions should allow protocols to be shared with colleagues without changing the underlying architecture.

---

# Drugs

Use **Drugs** instead of **Molecules** throughout the application.

Display a searchable alphabetical drug database.

Each drug should contain structured data such as:

* Name
* Category
* Species
* Clinical indication
* Dose
* Dose interval
* Dose range
* Administration route
* Frequency
* Concentration(s)
* Maximum dose
* Contraindications
* Warnings
* Notes

Species-specific dosing must be supported.

Different administration routes may have different recommended doses.

Different concentrations of the same drug must be supported.

Users should also be able to create, edit, and manage their own custom drugs.

---

# Drug Information

Drug information should be based on the most recent BSAVA Formulary available.

However, the application must **not** depend on the PDF itself.

Instead, use a structured database containing the necessary drug information.

The PDF should be treated only as a reference when building or updating the database.

The project owner may manually enter or import data from a properly licensed source into the application's own database.

The application should never function as a PDF reader.

* mL

* Infusions

* kg

# User Accounts

Authentication is required.

Each user should have a private account.

Personal data should remain isolated.

Protocols and custom drugs belong to the authenticated user.

The architecture should support future sharing of protocols with selected colleagues.

---

# Patient History

Patient history is not part of the MVP.

However, the architecture should allow future implementation without major redesign.

# General Principle

Whenever there is more than one possible solution, choose the one that is:

1. Simplest to understand.
2. Easiest to maintain.
3. Scalable.
4. Compatible with future React Native migration.
5. Suitable for a production-quality veterinary application.
